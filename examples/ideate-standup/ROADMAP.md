> **Product:** Product Understanding: Markdown Standup Digest CLI

This roadmap delivers the complete greenfield implementation of Markdown Standup Digest CLI — a self-contained Go binary that reads a TOML config, walks a designated Markdown notes folder to surface open GFM-style TODOs and recently-modified files, composes a human-readable standup digest, and HTTP-POSTs it to a Slack incoming webhook for cron-driven daily delivery. Nine tasks are sequenced from project scaffolding through four independently testable subsystems (config loading, folder scanning, TODO extraction, digest composition) to Slack delivery wiring, cross-platform binary packaging, and user-facing documentation, ensuring each task has verified predecessors and the build engine can execute them in strict dependency order without backtracking.

## Open questions

- TODO syntax scope: should the extractor recognise only GFM checkbox syntax (`- [ ] item`) or also freeform prefixes such as `TODO:` and `FIXME:`? The extraction logic and test corpus depend entirely on this definition. [brief:idea]
- Recency window: is the look-back period for 'recently changed files' a hardcoded 24 h constant, a configurable TOML field, or dynamically inferred from an optional last-run timestamp file? Behaviour on missed runs (weekends, holidays) is unspecified. [brief:idea]
- Config file discovery: should the binary look for the TOML file at a fixed XDG path (`~/.config/standupbot/config.toml`), the current working directory, or a path supplied via a `-config` CLI flag? [brief:idea]
- Slack message format: should the digest be posted as plain mrkdwn text or as a structured Slack Block Kit payload? These require different JSON schemas and have different size constraints. [brief:idea]
- Subdirectory handling: does the folder scan recurse into nested subdirectories, or process only the top-level `.md` files? [brief:idea]
- Webhook URL secrecy: should the TOML config support an env-var reference (e.g. `webhook_url = "$SLACK_WEBHOOK_URL"`) to avoid storing the secret in plaintext on disk? [brief:idea]
- Error handling on delivery failure: should a Slack POST failure cause the binary to exit non-zero (enabling cron `MAILTO` alerting), silently log to a local file, or retry with back-off? [brief:idea]
